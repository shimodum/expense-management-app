# 経費精算アプリ 要件定義書

## 1. プロジェクト概要

本プロジェクトは、Laravelを用いた経費精算Webアプリケーションを個人開発するものである。単にアプリケーションを完成させることだけを目的とせず、実務・チーム開発を意識した設計、Git運用、コードレビュー、ドキュメント管理を取り入れ、Claude Codeを活用したAI駆動開発の実践経験を積むことも目的とする。

## 2. 開発背景

経費精算業務は、多くの企業で紙媒体やExcelファイルによる運用が残っており、以下のような課題を抱えていることが多い。

- 申請状況(未提出/提出済み/承認済み/却下)が一覧で可視化されていない
- 申請内容や承認履歴の検索・管理が煩雑
- 承認者が申請内容を確認・承認/却下するまでのプロセスが属人化しやすい

このような業務課題は、CRUD操作に加えて「状態遷移(ステータス管理)」や「ロールに応じた認可制御」を伴う典型的な業務システムであり、Laravel学習および実務を意識した設計・開発の題材として適していると考え、本テーマを選定した。

## 3. 開発目的

- 実務・チーム開発を意識した設計、Gitを利用した実務を意識した開発フロー（ブランチ運用・コミット・Pull Requestを模擬的に実践する）、コードレビュー、ドキュメント管理を実践する
- Claude Codeを活用したAI駆動開発(要件定義・設計・実装・レビューの各フェーズでAIを活用する開発スタイル)の実践経験を積む
- Laravelにおける認証・認可、状態遷移(ステータス管理)を伴うアプリケーション構築力を習得する

## 4. システム概要

Laravel(MVCアーキテクチャ)によるWebアプリケーションとして構築する。認証機能を備え、一般社員が経費申請を登録してから、管理者が承認/却下するまでの一連のワークフローを提供する。

- フロントエンド: Bladeテンプレート(MVPではシンプルなサーバーサイドレンダリングを想定)
- バックエンド: Laravel(PHP)
- データベース: MySQL(前提条件・制約を参照)
- 認証: Laravel標準の認証機能を利用したログイン/ログアウト、ロールベースのアクセス制御

## 5. 想定利用者

- 一般社員
- 管理者

## 6. 利用者ごとの役割

### 6.1 一般社員

- 自分の経費申請の登録
- 自分の経費申請の閲覧
- 経費申請の編集(下書き状態、または却下された申請のみ)
- 経費申請の削除(下書き状態のみ)
- 経費申請の提出
- 却下された申請の修正・再提出

### 6.2 管理者

- 提出された経費申請の一覧・詳細閲覧
- 経費申請の承認
- 経費申請の却下(却下理由コメントの入力を含む、任意項目)

## 7. 利用シーン

1. 一般社員が出張・業務にかかった経費について、利用日・金額・カテゴリ・支払先・内容・領収書画像を登録し、内容を確認した上で申請を提出する。
2. 管理者が月末や随時のタイミングで提出済みの申請一覧を確認し、内容が適切なものは承認、不備があるものは却下する(必要に応じてコメントを付す)。
3. 却下された申請について、一般社員が却下理由を確認した上で内容を修正し、再度提出する。
4. 一般社員は自分の過去の申請(下書き/提出済み/承認済み/却下)をいつでも一覧・履歴として閲覧できる。

## 8. MVPの機能要件

### 8.1 認証機能

- ログイン/ログアウト
- ロール(一般社員/管理者)に応じたアクセス制御・画面表示の出し分け

### 8.2 経費申請機能(一般社員)

- 経費申請の登録(利用日、金額、経費カテゴリ、支払先、内容、領収書画像)
- 自分が登録した経費申請の一覧・詳細閲覧
- 経費申請の編集(下書き状態、または却下された申請のみ)
- 経費申請の削除(下書き状態のみ)
- 経費申請の提出(ステータスを「下書き」→「提出済み」に変更)
- 却下された申請の再提出(ステータスを「却下」→「提出済み」に変更)

### 8.3 承認機能(管理者)

- 提出済み経費申請の一覧・詳細閲覧
- 経費申請の承認(ステータスを「提出済み」→「承認済み」に変更)
- 経費申請の却下(ステータスを「提出済み」→「却下」に変更、却下理由コメントは任意入力)

### 8.4 ステータス管理

経費申請は以下のいずれかのステータスを持つ。

| ステータス | 説明 |
|---|---|
| 下書き | 一般社員が登録したが未提出の状態。編集・削除が可能 |
| 提出済み | 一般社員が提出し、管理者の確認待ちの状態。編集・削除不可 |
| 承認済み | 管理者が承認した状態。編集・削除不可 |
| 却下 | 管理者が却下した状態。一般社員は編集して再提出可能 |

## 9. 業務ルール

- 経費申請を編集・削除できるのは「下書き」状態のときのみ。ただし却下された申請は編集のみ可能(削除は不可とし、再提出を促す)。
- 「提出済み」「承認済み」の申請は一般社員による編集・削除を一切不可とする。
- 却下された申請を編集した場合、ステータスは「提出済み」に戻り、再度管理者の確認対象となる。
- 金額は0より大きい整数(円単位)で入力しなければならない。
- 経費カテゴリは、あらかじめ用意されたマスタ(例: 交通費、宿泊費、交際費、消耗品費、その他)から選択する。
- 領収書画像の添付はMVPでは任意項目とする(将来的に必須化を検討する)。
- 一般社員は自分自身が登録した申請以外を閲覧・操作できない。
- 管理者は全社員の申請を閲覧・承認/却下できるが、自ら経費申請を登録する機能はMVPでは対象外とする。

## 10. 非機能要件

- **認可制御**: ロール(一般社員/管理者)に基づくアクセス制御を、Laravelの認可機能(Policy/Gate等)を用いて実装する
- **バリデーション**: フォームリクエスト等を用いて入力値の妥当性を検証する
- **ログ記録**: Laravel標準ログのみ
- **テスト**: 主要なユースケース(申請登録・提出・承認・却下)についてFeatureテストを整備する
- **コーディング規約**: PSR-12に準拠したコーディングスタイルを維持する
- **Git運用**: ブランチ運用ルール、コミットメッセージ規約、プルリクエストベースの開発フローを定めて運用する
- **ドキュメント管理**: 要件定義書・設計書等をリポジトリ内(`docs/`)で管理し、変更履歴をGitで追跡する

## 11. 今回の開発対象外

- 多段階承認フロー(複数承認者による段階的な承認)
- メール・Slack等の通知機能
- 会計システム・外部システムとの連携
- 多通貨対応
- 部署・組織階層の管理
- PDF形式の領収書対応(画像形式のみ対応)
- 外部ストレージ(S3等)への領収書保存
- モバイルアプリ化
- 経費上限・予算管理機能
- 管理者自身による経費申請機能

## 12. 将来的な拡張候補

- 承認フローの多段階化(例: 上長承認→経理承認)
- メール・Slack等による申請/承認/却下時の通知機能
- 経費データのCSVエクスポート
- カテゴリ別・部署別の経費上限アラート
- 部署別・期間別の集計ダッシュボード
- PDF形式の領収書対応
- 外部ストレージ(S3等)への領収書保存

## 13. 前提条件・制約

- 本プロジェクトではClaude CodeおよびChatGPTを開発支援ツールとして利用する
- 開発言語・フレームワーク: PHP / Laravel(最新の安定バージョンを使用)
- データベース: MySQL
- 個人開発であり、単一の開発環境(ローカル環境)での動作を前提とする。本番環境への運用・デプロイは想定しない
- チーム開発を想定した運用(Git運用・レビュー等)は、単独での開発であっても実務同様のプロセスを模して実施する
- 開発期間は目安として設定し、要件定義・設計・実装・テストの各フェーズを段階的に進める

## 14. 用語定義

| 用語 | 説明 |
|---|---|
| 申請 | 一般社員が登録する経費精算のための1件のデータ(経費申請) |
| 下書き | 申請が登録されたが、まだ提出されていない状態 |
| 提出 | 一般社員が申請内容を確定し、管理者の確認対象とする操作 |
| 承認 | 管理者が申請内容を認め、精算対象として確定する操作 |
| 却下 | 管理者が申請内容に不備・不適切な点があるとして差し戻す操作 |
| 経費カテゴリ | 経費の種別を分類するための区分(交通費、宿泊費等) |
| 領収書 | 経費の証憑として添付する画像ファイル |
